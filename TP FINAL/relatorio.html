Nexus Chat: Sistema de Chat Distribuído
Plataforma de comunicação em tempo real com arquitetura de microsserviços
Aluna: Ester Morais Neves Data: Junho / 2026 Repositório: https://github.com/estermorais/Sistemas-Distribuidos/tree/tp-1-e-2

1. Introdução e Objetivos

O objetivo deste trabalho foi desenvolver o Nexus Chat, um sistema de chat em tempo real usando arquitetura de microsserviços. A ideia era que vários usuários pudessem se comunicar de forma confiável, tanto em canais públicos (1:N) quanto em conversas privadas (1:1), com as mensagens chegando de forma instantânea.

Para isso, três requisitos principais guiaram o projeto: manter o sistema disponível mesmo que um componente falhe, garantir baixa latência na entrega das mensagens e conseguir adicionar mais servidores conforme o número de usuários cresce. Os dois primeiros foram atendidos com WebSockets e a separação em serviços independentes; o terceiro foi resolvido usando Redis Pub/Sub para sincronizar as réplicas do Chat Service.

2. Arquitetura do Sistema

O sistema foi dividido em dois microsserviços independentes, cada um rodando em container Docker separado. Um Nginx na frente recebe todas as requisições e distribui para o serviço correto, funcionando também como balanceador de carga entre as duas réplicas do Chat Service. O Docker Compose orquestra a inicialização e a comunicação entre todos os componentes.

Como funciona o fluxo

O browser se conecta sempre ao Nginx na porta 80. Requisições para /api/auth/ são redirecionadas ao Auth Service; requisições para /api/chat/ e conexões WebSocket (/ws/) vão para uma das duas réplicas do Chat Service via round-robin. Quando uma réplica recebe uma mensagem, ela publica no canal Redis; a outra réplica está inscrita nesse canal e repassa para seus usuários conectados.

3. Implementação e Decisões Tomadas

3.1 Auth Service

Este serviço cuida de tudo relacionado a usuário: cadastro, login e listagem de contatos. Quando o usuário faz login, o sistema gera um token JWT válido por 60 minutos com o user_id e o nome do usuário. As senhas nunca são guardadas em texto puro; o bcrypt gera um hash diferente a cada cadastro, o que evita problemas com rainbow table. Os endpoints disponíveis são POST /register, POST /login, GET /users e GET /verify.

3.2 Chat Service e WebSockets

Cada usuário abre uma conexão WebSocket na rota /ws/{user_id}?token=JWT. O token é verificado logo no início da conexão; se for inválido, a conexão é recusada. Internamente, uma classe chamada ConnectionManager guarda as conexões abertas e usa o Redis Pub/Sub para repassar as mensagens. Isso foi importante porque, com duas réplicas do serviço rodando ao mesmo tempo, uma mensagem recebida por uma réplica precisa chegar aos usuários conectados na outra também.

Problema encontrado e resolvido: na primeira versão, cada mensagem chegava duplicada no chat. Isso acontecia porque o código entregava a mensagem direto na conexão local e também pelo Redis ao mesmo tempo. A solução foi fazer tudo passar pelo Redis, deixando a entrega local só como fallback quando o Redis não está disponível.

3.3 Persistência das mensagens

As mensagens ficam salvas na tabela messages do PostgreSQL. Para mensagens privadas, o campo recipient_id guarda o destinatário; para canais públicos, esse campo fica nulo e o room_id identifica o canal. O histórico pode ser consultado pelos endpoints GET /messages?room_id=X para canais e GET /messages/private/{id} para conversas diretas.

3.4 Frontend

A interface foi feita com HTML, CSS e JavaScript puro, sem nenhum framework. O tema é escuro com cores em tons de roxo e índigo. Quando a conexão WebSocket cai, o sistema tenta reconectar automaticamente a cada 3 segundos. As mensagens não lidas mostram um badge numérico na lista de conversas, e a sessão é restaurada do sessionStorage se o usuário recarregar a página sem ter feito logout.

3.5 Criação de Salas

O sistema suporta a criação de salas de forma dinâmica. Ao iniciar, o Chat Service verifica se as salas padrão (geral, tecnologia, off-topic) existem no banco e as cria automaticamente caso não existam. Além disso, qualquer usuário autenticado pode criar novas salas pelo botão "+" na barra lateral. O nome é normalizado automaticamente (espaços viram hífens, tudo em minúsculo) e nomes duplicados são rejeitados com erro 409. As salas ficam salvas na tabela rooms do PostgreSQL e são carregadas dinamicamente pelo frontend a cada login.

3.6 Exclusão de Conta

Um usuário pode excluir sua própria conta através do botão "Excluir conta" na parte inferior da barra lateral. A exclusão é feita pelo endpoint DELETE /users/me no Auth Service e funciona como um soft delete: o campo is_active do usuário é marcado como false no banco, mas o registro permanece para manter o histórico de mensagens consistente. Após a exclusão, a sessão é encerrada automaticamente e o usuário não consegue mais fazer login com aquela conta.

4.1 Testes Unitários

Os testes unitários foram feitos para verificar as partes mais críticas de cada serviço de forma isolada, sem depender de banco de dados ou rede. No Auth Service, foram testados o hash de senhas com bcrypt (incluindo casos de senha vazia e senha errada), a geração do token JWT e a validação dos campos dentro do token. No Chat Service, foram testados a decodificação do JWT (token válido, adulterado e com secret errado) e se o campo is_private das mensagens estava sendo setado corretamente.

4.2 Testes de Integração

Para os testes de integração, foi usado um banco SQLite em memória no lugar do PostgreSQL, aproveitando o sistema de injeção de dependência do FastAPI para trocar o banco sem mudar o código do serviço. Isso permitiu rodar todos os testes sem precisar de Docker ou banco externo instalado.

Os cenários testados incluíram: cadastro de usuário novo, tentativa de cadastro com nome duplicado, senha muito curta; login correto, login com senha errada e usuário inexistente; acesso às rotas protegidas com e sem token; salvar mensagens públicas e privadas no banco; paginação do histórico; e conexão WebSocket com token inválido, envio de mensagem para canal e envio de mensagem privada.

4.3 Teste de Carga

O teste de carga foi feito com um script Python que usa a biblioteca websockets para simular vários usuários ao mesmo tempo. O script registra os usuários, faz login de todos em paralelo, aguarda todos estarem conectados via WebSocket e então dispara o envio simultâneo. Abaixo o resultado com 10 usuários e 5 mensagens cada:

RESULTADO FINAL: PASSOU
As 10 conexões simultâneas foram atendidas sem nenhum erro. O tempo médio de login ficou em 18 ms e o sistema processou 15 mensagens por segundo. O número de mensagens recebidas (487) ser maior que o enviado (50) é esperado porque cada mensagem de canal chega para todos os usuários conectados naquele momento.

5. Conclusão

O trabalho atendeu aos requisitos pedidos e foi além: dois microsserviços independentes (Auth e Chat), comunicação em tempo real via WebSockets, persistência no PostgreSQL, interface funcional e ainda funcionalidades extras como criação dinâmica de salas e exclusão de conta. A parte que exigiu mais atenção foi o Redis Pub/Sub, que resolveu o problema de múltiplas réplicas do Chat Service sem precisar centralizar as conexões em um único processo.

No total foram 33 testes automatizados cobrindo as funcionalidades principais. O teste de carga confirmou que o sistema aguenta os 10 usuários simultâneos e ainda tem espaço para crescer. O código está disponível no repositório github.com/estermorais/Sistemas-Distribuidos na branch tp-1-e-2. Como melhoria futura, seria interessante adicionar confirmação de entrega de mensagem e um mecanismo de reconexão com recuperação do histórico não recebido durante a queda.
