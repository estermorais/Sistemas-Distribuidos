O que foi construído

Visão Geral do Projeto

  • Chat em tempo real via WebSockets — mensagens chegam instantaneamente sem polling
  • Canais públicos (1:N) — salas criadas dinamicamente por qualquer usuário
  • Mensagens diretas (1:1) — conversas privadas entre dois usuários
  • Histórico persistido — mensagens salvas no banco, novos usuários veem conversas anteriores
  • Autenticação JWT — token gerado no login, válido 60 min, verificado em cada conexão
  • Escalabilidade horizontal — 2 réplicas do Chat Service sincronizadas via Redis
  • Criar salas — qualquer usuário cria um canal novo pelo botão "+"
  • Excluir conta — soft delete via DELETE /users/me
  • 33 testes automatizados — unitários + integração sem Docker
  • Reconexão automática — WebSocket reconecta sozinho após queda

Arquitetura de microsserviços

Sim, estamos usando Microsserviços

🔐 Auth Service
  • Container próprio, processo independente
  • Responsabilidade única: usuários e tokens
  • POST /register · POST /login
  • GET /users · DELETE /users/me
  • Pode escalar independentemente
💬 Chat Service
  • Container próprio, roda com 2 réplicas
  • Responsabilidade única: mensagens e conexões
  • WebSocket /ws/{user_id}
  • GET/POST /rooms · GET /messages
  • Redis Pub/Sub sincroniza as réplicas

Cada serviço tem seu próprio ciclo de vida. Se o Auth Service cair, o chat continua funcionando para quem já está autenticado. Se uma réplica do Chat Service reiniciar, a outra absorve o tráfego.

Como eles se comunicam?
  • Frontend → Nginx → Auth Service via HTTP REST
  • Frontend → Nginx → Chat Service via WebSocket
  • Chat Service ↔ Redis via Pub/Sub (não há chamada HTTP entre serviços)
  • Ambos → PostgreSQL via SQLAlchemy (tabelas separadas)
Segurança e tempo real

Autenticação e Comunicação

🔐 Como a autenticação funciona

1 — Cadastro
Senha enviada via HTTPS → bcrypt gera hash (custo 12) → salvo no PostgreSQL. A senha em texto puro nunca é persistida.
2 — Login
bcrypt.verify() compara senha com hash → gera JWT HS256 com user_id e username, validade 60 min.
3 — Uso
JWT enviado em cada requisição. WebSocket: token na query string. Chat Service valida o token antes de aceitar a conexão.

📡 Como as mensagens chegam

  • WebSocket é assíncrono por natureza — a mensagem chega quando o servidor envia, sem o cliente precisar perguntar (sem polling)
  • Pub/Sub: quando uma réplica recebe a mensagem, publica no Redis; todas as réplicas inscritas entregam para seus usuários conectados
  • Sem duplicação: tudo passa pelo Redis; a entrega local é só fallback quando Redis não está disponível
  • Sim, as mensagens são assíncronas — você envia e o destinatário recebe quando a mensagem chega, independentemente de quando ele vai ver
O que o usuário pode fazer

Funcionalidades Implementadas

🏠
Criar e entrar em salas
Salas padrão (geral, tecnologia, off-topic) criadas automaticamente no startup. Qualquer usuário pode criar novas salas pelo botão "+" — o nome é normalizado e salvo no banco.
💬
Selecionar destinatário
Na barra lateral: clicar em um canal (#) envia para todos no canal (1:N). Clicar em um contato abre conversa privada (1:1). O frontend envia o campo room_id ou recipient_id no JSON.
📜
Histórico para novos usuários
Ao abrir qualquer canal, o frontend busca os últimos 60 mensagens via GET /messages?room_id=X. Novos usuários veem toda a conversa anterior ao seu cadastro.
🗑️
Excluir conta
Botão "Excluir conta" na base da sidebar chama DELETE /users/me. É um soft delete: campo is_active = false no banco. O usuário não consegue mais fazer login, mas as mensagens dele continuam no histórico.
Presença e sinalização

Status Online

✅ O que está implementado
  • Ponto verde no cabeçalho quando o WebSocket está ativo
  • Ponto vermelho e texto "reconectando..." quando a conexão cai
  • Reconexão automática a cada 3 segundos em caso de queda
  • Badge numérico de mensagens não lidas por canal/DM
⚠️ O que não está (ainda)
  • Lista global de "quem está online agora" visível para todos
  • Confirmação de leitura ("mensagem lida")

Para implementar uma lista global de online, seria necessário emitir um evento user:online no Redis quando o WebSocket conecta e um user:offline quando desconecta, e o frontend assinaria esses eventos para atualizar a lista.

A arquitetura atual já suporta isso — seria uma extensão natural do sistema de Pub/Sub já existente, sem necessidade de mudança na infraestrutura.
